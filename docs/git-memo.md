# ソフトウェア工学 2024

講義で学んだことをmarkdown形式でまとめる。  

## ソフトウェア
### ソフトウェアの定義
1. 実行されることによって必要な特性、機能、性能を提供する命令語群(コンピュータプログラム)
2. プログラムが適切に情報を扱うことを可能とするデータ構造
3. プログラムの操作や使用法を記述した情報
### ソフトウェア工学とは
ソフトウェア工学とは、ソフトウェアの開発、運用、メンテナンス等に対してエンジニアリングを適用する学問分野であり、ソフトウェアシステムが変化し続ける状況に対応する方法論を考え出すことを目的とする。具体的には、品質・コスト・納期の最適なバランスを実現するための手法・方法論などを研究する。  
### ソフトウェア工学の必要性
今日の社会にはソフトウェアの存在が不可欠であり、高品質なソフトウェアを効率よく開発し、メンテナンスを行う必要がある。特に、ライフラインやインフラストラクチャーとしてのソフトウェアも増加しており、不測の事態に対応した事業・サービスの継続が求められる。また、ソフトウェアの利用の仕方は多様化しており、今後も変化していくと考えられるため、ニーズに対応したソフトウェアの開発が課題となる。

## ソフトウェアライフサイクル
ソフトウェアライフサイクルとは、ソフトウェアを開発して、使用し、破棄するまでの一連の工程のことである。その工程は、計画、要件定義、設計、製作、テスト、運用・保守などに分けられる。  
一連の流れとしては、まず計画でステークホルダーから意見を集め、要件の収集・分析を行い、要件定義で実現する機能を明確にする。設計の段階で、要件定義書を元にコーディングを行うための設計書を作成し、制作では設計書に従ってシステムを構築する。その後、実装したソフトウェアが正しく動作するかを確認するテストを実施し、バグの発見・対処を行う。運用・保守では、実際にシステムを稼働し、継続的なモニタリングや不具合の対処などを行う。

## プロジェクト
プロジェクトとは、期間と目的が定められた業務のことである。プロジェクトは、目標を明確に設定し、目標に向けての筋道を検討するバックキャスティングという計画手法に基づいている。

## ソフトウェア分析
ソフトウェアを評価する指標には、以下のようなものがある。
1. コードの物量  
　ソースコードの行数(ステップ数)はソフトウェアの規模を測る指標となる。  
　オブジェクト容量に関しては、組み込み型ソフトウェアはサイズが小さいほど良いとされる。
2. バグ発生率  
　バグ発生率は、1ステップ数あたりの発生バグ数である。
3. ファンクションポイント法  
　機能の数や難易度を点数化することで、システム規模を概算することができる。
4. 使い勝手  
　画面の視認性や操作性などの要素もソフトウェアを評価する指標となる。

## 開発プロセス
### ウォーターフォール型開発プロセス
要件定義、設計、制作、テストといった各工程を、上流工程から下流工程へ順に行っていく手法である。  
進捗管理が容易で、成果物が明確になるが、後工程にしわ寄せが集中するリスクがある。
### スパイラルモデル
プログラム開発を小さなフェーズに分割し、フェーズごとにプロトタイプを作成し、フィードバックを得るという工程をフェーズごとに行う手法である。ただし、プロトタイプ作成に想定外の作業量が発生するリスクがある。
### アジャイルプロセス
システムを機能単位で分割し、計画・設計・実装・テストといった開発工程を繰り返す手法である。  
開発期間を短縮でき、仕様変更にも柔軟に対応できるという利点がある。

## WBS
WBSとは、プロジェクト目標を達成し、必要な要素成果物を生成するために、プロジェクト全体の作業を階層的に分解した構成図である。
### WBSの作り方
- スコープを明確にする
- 大きな作業をグルーピングする
- グルーピングした作業の相互関連を考える
- 各グループの作業を洗い出す

## UML
UMLとは、オブジェクト指向でシステムの仕様を分析、設計、記述するために、標準規格として作られた図の記法である。
### UMLの種類
- ユースケース図  
　利用者視点でシステムがどのように利用されるかを表した図である。
　アクションを起こすアクターとシステムのふるまいを表すユースケース、ユースケース間の関連といった要素から構成される。
- アクティビティ図  
　システムの一連の流れや状態遷移を表した図である。
　アクションノード、矢印、初期ノード、最終ノード、フォーク、ジョインといった要素から構成される。
- クラス図  
　クラスの定義や関連付けを表す図である。
　各クラスには属性、操作、アクセスが可能な範囲を表現する可視性を持たせる。
- オブジェクト図  
　クラスが実体化したインスタンスの関係性を表した図である。
- シークエンス図  
　処理の流れを時間軸に沿って表した図である。
- コンポーネント図  
　複数のクラスで構成される処理の内部構造や相互作用を表した図である。

## コーディング
ソフトウェアの開発において読みやすいコードを書くことは必須であり、PEP8のようなガイドラインが定められている。ルールの例は以下の通りである。  
- 1行の長さは79文字以内にする
- インデントはスペース4つを使う
- 演算子の前後にスペースを入れる
- 改行では演算子の位置を揃える
- importは標準ライブラリ、サブパーティ関連、ローカルなライブラリに特有のものを順に分けて記述する
- 他者にとってわかりやすいコメントを記載する
- 命名規則に従う(クラスはCapWords方式、関数や変数は小文字のみなど)

## Gitの基本
### Gitとは
Gitとは、分散型のバージョン管理システムである。  
バージョン管理とは、ファイルに対して「いつ」「誰が」「どのように」変更したのかを管理することである。  
Gitを使うことで、古いバージョンから新しいバージョンのファイルまで一元的に管理することが可能であり、各ファイルの変更履歴も記録されるため、古いバージョンに戻すことも容易である。  
また、複数人でファイルや変更履歴を共有することも可能なため、他者と共同で編集を行うことができる。  

- コミット : ファイル作成・変更・削除の記録
- レポジトリ : gitが管理するプロジェクトのフォルダ
- ブランチ : 作業を枝分かれさせる機能

### 基本のコマンド
- git init : gitの初期化・設定開始
- git status : ワークツリーのステータスを表示
- git log : ログを表示
- git add : ファイルをステージングエリアに追加する
- git commit : コミットの実行
- git reset : コミットを取り消す
- git clone : レポジトリをコピー
- git pull : リモートレポジトリの同期
- git push : 変更をアップロードする
- git branch : ブランチの作成
- git checkout : ブランチの切り替え
- git merge : ブランチの統合

## CI/CD
### Continuous Integration (CI)
コード変更を共有レポジトリに頻繁に統合するプロセスである。自動テストとビルドを定期的に実行することで、バグの早期発見と修正を可能にする。CIの自動化により、開発のスムーズな進行を促進できる。
### Continuous Delivery (CD)
コード変更をテスト環境や本番環境に自動的にデプロイするプロセスである。自動デプロイメントを組み込むことで、手動操作によるデプロイの必要性を排除できる。CDにより、ユーザーフィードバックを迅速に反映することが可能になる。
